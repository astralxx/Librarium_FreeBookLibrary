# Анализ ошибок и проблем в backend-коде library-reports

## Executive Summary

Проведен глубокий анализ backend-кода проекта library-reports. Выявлены критические проблемы безопасности, архитектурные недостатки и баги, требующие немедленного исправления.

---

## 1. КРИТИЧЕСКИЕ ПРОБЛЕМЫ БЕЗОПАСНОСТИ

### 1.1 SQL-инъекции (КРИТИЧЕСКИЙ УРОВЕНЬ)

**Статус**: ✅ Частично защищено (используются параметризованные запросы)

**Детали**:
- Все SQL-запросы используют параметризованные запросы (`?`)
- Однако, в [`books-service/main.go:239`](books-service/main.go:239) используется `CONCAT` с пользовательским вводом, что безопасно благодаря параметризации

**Рекомендации**: Добавить дополнительную валидацию входных данных

### 1.2 Отсутствие валидации входных данных (КРИТИЧЕСКИЙ УРОВЕНЬ)

**Проблемы**:

#### auth-service/main.go:
1. **Отсутствие валидации email** (строка 154-157):
   ```go
   var user User
   if err := json.NewDecoder(r.Body).Decode(&user); err != nil {
       http.Error(w, "Invalid JSON", http.StatusBadRequest)
       return
   }
   ```
   - Нет проверки формата email
   - Нет проверки длины email
   - Нет санитизации

2. **Отсутствие валидации username** (строка 159-163):
   ```go
   if user.Password == "" {
       http.Error(w, "Password cannot be empty", http.StatusBadRequest)
       return
   }
   ```
   - Нет проверки длины username
   - Нет проверки допустимых символов
   - Нет проверки на SQL-инъекции в username

3. **Слабая валидация пароля** (строка 159-163):
   - Проверяется только на пустоту
   - Нет минимальной длины
   - Нет требований к сложности
   - Нет максимальной длины (DoS через очень длинный пароль)

#### books-service/main.go:
1. **Отсутствие валидации ID книги** (строка 190-191):
   ```go
   vars := mux.Vars(r)
   id := vars["id"]
   ```
   - Нет проверки что ID является числом
   - Нет проверки на положительное значение

2. **Отсутствие валидации поискового запроса** (строка 226-231):
   ```go
   searchTerm := query.Get("q")
   if searchTerm == "" {
       http.Error(w, "Search term is required", http.StatusBadRequest)
       return
   }
   ```
   - Нет проверки длины
   - Нет санитизации от XSS
   - Нет ограничения на специальные символы

### 1.3 Утечки чувствительных данных в логах (ВЫСОКИЙ УРОВЕНЬ)

**Проблемы**:

1. **auth-service/main.go:210**:
   ```go
   log.Printf("User registered: %s", user.Username)
   ```
   - Логируется username (потенциально чувствительная информация)

2. **auth-service/main.go:266**:
   ```go
   log.Printf("User logged in: %s", dbUser.Username)
   ```
   - Логируется username при входе

3. **auth-service/main.go:174, 188, 195**:
   ```go
   log.Printf("Database error: %v", err)
   log.Printf("Password hashing error: %v", err)
   log.Printf("SQL error: %v", err)
   ```
   - Детали ошибок БД могут содержать чувствительную информацию

**Рекомендации**:
- Не логировать username в production
- Использовать структурированное логирование
- Маскировать чувствительные данные

### 1.4 Отсутствие Rate Limiting (КРИТИЧЕСКИЙ УРОВЕНЬ)

**Проблемы**:
- Нет защиты от brute-force атак на `/login`
- Нет защиты от brute-force атак на `/register`
- Нет ограничения на количество запросов к API
- Нет защиты от DDoS атак

**Рекомендации**:
- Реализовать rate limiting на уровне gateway
- Использовать token bucket или sliding window алгоритм
- Добавить CAPTCHA после нескольких неудачных попыток

### 1.5 Слабая JWT аутентификация (ВЫСОКИЙ УРОВЕНЬ)

**Проблемы**:

1. **Отсутствие issuer и audience** (auth-service/main.go:436-442):
   ```go
   claims := &Claims{
       Username: username,
       Email:    email,
       RegisteredClaims: jwt.RegisteredClaims{
           ExpiresAt: jwt.NewNumericDate(expirationTime),
       },
   }
   ```
   - Нет Issuer (iss)
   - Нет Audience (aud)
   - Нет Subject (sub)
   - Нет JWT ID (jti) для предотвращения replay атак

2. **Отсутствие механизма отзыва токенов**:
   - Нет blacklist отозванных токенов
   - Нет возможности принудительного выхода
   - Нет проверки на отзыв при каждом запросе

3. **Длительное время жизни токена** (24 часа):
   - Слишком длинное время жизни для access token
   - Нет refresh token механизма

### 1.6 Отсутствие CSRF защиты (ВЫСОКИЙ УРОВЕНЬ)

**Проблемы**:
- Нет CSRF токенов для POST/PUT/DELETE запросов
- Cookie не имеет SameSite в некоторых местах
- Нет проверки Origin/Referer заголовков

### 1.7 Небезопасное хранение секретов (СРЕДНИЙ УРОВЕНЬ)

**Проблемы**:
- JWT_SECRET должен быть более сложным (минимум 256 бит)
- Нет ротации секретов
- .env файлы не защищены от несанкционированного доступа

### 1.8 CORS конфигурация (СРЕДНИЙ УРОВЕНЬ)

**Проблемы** (gateway/main.go:170-196):
- `Access-Control-Allow-Credentials: true` с динамическим origin
- Нет строгой проверки origin
- Нет ограничения на методы и заголовки

## 2. АРХИТЕКТУРНЫЕ ПРОБЛЕМЫ

### 2.1 Отсутствие слоистой архитектуры

**Текущее состояние**: Монолитные обработчики
```go
func registerHandler(w http.ResponseWriter, r *http.Request) {
    // 1. Декодирование JSON (presentation)
    // 2. Валидация (business logic)
    // 3. Проверка существования (data access)
    // 4. Хэширование пароля (business logic)
    // 5. Сохранение в БД (data access)
    // 6. Генерация токена (business logic)
    // 7. Установка cookie (presentation)
    // 8. Ответ (presentation)
}
```

**Проблемы**:
- Смешение логики разных слоев
- Сложно тестировать
- Сложно модифицировать
- Нарушение принципа единственной ответственности

### 2.2 Дублирование кода

**Примеры**:

1. **Функция getEnv** (дублируется в 3 файлах):
   - auth-service/main.go:79-84
   - books-service/main.go:47-52
   - gateway/main.go:52-57

2. **Настройка graceful shutdown** (дублируется в 3 файлах):
   - auth-service/main.go:130-148
   - books-service/main.go:105-123
   - gateway/main.go:209-227

3. **Обработка ошибок БД** (дублируется):
   - Повторяющиеся паттерны обработки sql.ErrNoRows
   - Повторяющиеся паттерны логирования ошибок

4. **Сканирование книг** (books-service/main.go:135-147):
   - Функция scanBook вызывается в 5 местах
   - Можно вынести в репозиторий

### 2.3 Отсутствие абстракций

**Проблемы**:
- Нет интерфейсов для работы с БД
- Нет сервисного слоя
- Нет репозиториев
- Нет middleware для общих функций

### 2.4 Нарушение принципов SOLID

1. **Single Responsibility Principle (SRP)**:
   - Обработчики делают слишком много
   - Смешение валидации, бизнес-логики и работы с БД

2. **Open/Closed Principle (OCP)**:
   - Добавление нового endpoint требует изменения существующего кода
   - Нет плагинной архитектуры

3. **Liskov Substitution Principle (LSP)**:
   - Нет интерфейсов для подмены реализаций

4. **Interface Segregation Principle (ISP)**:
   - Нет интерфейсов вообще

5. **Dependency Inversion Principle (DIP)**:
   - Зависимости от конкретных реализаций БД
   - Нет инъекции зависимостей

## 3. БАГИ И ЛОГИЧЕСКИЕ ОШИБКИ

### 3.1 Дублирование установки cookie (auth-service/main.go:380-383)

```go
// Устанавливаем cookie
setAuthCookie(w, tokenString)

// Устанавливаем cookie с токеном
setAuthCookie(w, tokenString)
```

**Проблема**: Cookie устанавливается дважды
**Влияние**: Избыточные заголовки ответа

### 3.2 Неправильный статус ответа при логине (auth-service/main.go:227)

```go
w.WriteHeader(http.StatusCreated)
```

**Проблема**: Статус 201 Created используется для логина
**Должно быть**: 200 OK
**Влияние**: Некорректная семантика HTTP

### 3.3 Отсутствие проверки на пустой массив книг (books-service/main.go:307-309)

```go
if len(books) == 0 {
    books = []BookResponse{}
}
```

**Проблема**: Проверка выполняется после обработки всех строк
**Должно быть**: Инициализировать books как пустой массив в начале

### 3.4 Неправильная обработка offset (books-service/main.go:359-368)

```go
offset := 0
if offsetStr := r.URL.Query().Get("offset"); offsetStr != "" {
    parsedOffset, err := strconv.Atoi(offsetStr)
    if err != nil {
        log.Printf("Invalid offset parameter: %v", err)
        http.Error(w, "Invalid offset parameter", http.StatusBadRequest)
        return
    }
    offset = parsedOffset
}
```

**Проблема**: Нет проверки на отрицательные значения
**Влияние**: Возможность SQL-инъекции через отрицательный offset

### 3.5 Неправильная маршрутизация в gateway (gateway/main.go:85-88)

```go
r.PathPrefix("/auth/").HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
    r.URL.Path = strings.TrimPrefix(r.URL.Path, "/auth")
    authProxy.ServeHTTP(w, r)
})
```

**Проблема**: Модификация r.URL.Path напрямую
**Лучше использовать**: PathRewrite в ReverseProxy

### 3.6 Отсутствие проверки на nil при сканировании (books-service/main.go:139-143)

```go
err := rows.Scan(&b.ID, &b.Title, &b.Author, &b.Genre,
    &b.CreatedAt, &imgData, &b.Type, &b.Feature)
if err != nil {
    return nil, err
}
```

**Проблема**: Нет проверки на NULL значения из БД
**Влияние**: Паника при NULL в обязательных полях

### 3.7 Неправильная обработка Google OAuth (auth-service/main.go:350-352)

```go
username := strings.Split(userInfo.Email, "@")[0]
_, err = db.Exec("INSERT INTO users (username, email, password) VALUES (?, ?, ?)",
    username, userInfo.Email, "google_auth")
```

**Проблемы**:
1. Username может содержать недопустимые символы
2. Нет проверки на уникальность username
3. Пароль "google_auth" - небезопасно

### 3.8 Отсутствие валидации JWT в gateway (gateway/main.go:253-265)

```go
func validateJWT(tokenString string) bool {
    claims := &jwt.RegisteredClaims{}
    token, err := jwt.ParseWithClaims(tokenString, claims, func(token *jwt.Token) (interface{}, error) {
        return jwtKey, nil
    })
    
    if err != nil {
        log.Printf("JWT validation error: %v", err)
        return false
    }
    
    return token.Valid
}
```

**Проблемы**:
1. Не проверяется issuer
2. Не проверяется audience
3. Не проверяется expiration (только встроенный механизм)
4. Не проверяется algorithm (возможна algorithm confusion атака)

## 4. ПРОБЛЕМЫ С БАЗОЙ ДАННЫХ

### 4.1 Отсутствие индексов

**Проблема**: Нет индексов для часто используемых полей
**Влияние**: Медленные запросы при большом количестве данных

**Рекомендации**:
```sql
CREATE INDEX idx_books_title ON books(title);
CREATE INDEX idx_books_author_id ON books(author_id);
CREATE INDEX idx_books_genre_id ON books(genre_id);
CREATE INDEX idx_books_feature ON books(feature);
CREATE INDEX idx_books_type ON books(type);
CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_users_username ON users(username);
```

### 4.2 Отсутствие каскадного удаления

**Проблема**: Нет ON DELETE CASCADE для внешних ключей
**Влияние**: Невозможность удаления авторов/жанров с книгами

### 4.3 Отсутствие soft delete

**Проблема**: Нет механизма мягкого удаления
**Влияние**: Потеря данных при удалении

### 4.4 Отсутствие версионирования схемы

**Проблема**: Нет миграций
**Влияние**: Сложность обновления схемы БД

### 4.5 Отсутствие схемы для auth-service

**Проблема**: init-auth.sql пуст
**Влияние**: Нет таблицы users

## 6. ПРОБЛЕМЫ С ЛОГИРОВАНИЕМ

### 6.1 Неинформативные логи

**Проблема**: Логи не содержат контекста
**Пример**:
```go

```

**Должно быть**:
```go
log.Printf("Database error in registerHandler: user=%s, error=%v", user.Username, err)
```

### 6.2 Отсутствие структурированного логирования

**Проблема**: Используется стандартный log пакет
**Рекомендация**: Использовать zap или logrus

### 6.3 Отсутствие уровней логирования

**Проблема**: Все логи одного уровня
**Рекомендация**: DEBUG, INFO, WARN, ERROR
